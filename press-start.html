Press Start Database Design

A team database project built around a real retail game store case, with a focus on requirements, normalization, and designing for actual store operations.

Role
Team project
Main focus
Requirements, normalization, ERD design
What it taught me
Designing around real business processes

Overview

For this project, my group designed a database system for Press Start, a game store dealing with inventory, trade-ins, repairs, quotes, pricing, and customer history. The case started from a messy current state with an Access inventory database, paper repair slips, no centralized tracking, inconsistent pricing, and no support for multiple locations.

What I liked about this one is that it was not just about drawing tables. It forced us to think about how the store actually works and what data the system would need in order to support that workflow without becoming disorganized.

Why this project is here:
I chose this project because it shows an important part of my development through the CPA program, whether that was building a complete system, improving how I structure code, understanding business requirements better, or learning from earlier mistakes.

What I worked on

I contributed to the reasoning behind the database structure and spent time thinking through whether the design actually matched a real retail environment.

A big part of that was the ERD and the relationships between customers, inventory, sales, repairs, employees, and pricing history. I also contributed to the conversations around scope, system boundaries, and how the design should support current needs without becoming unrealistic.

What the system needed to support

The project requirements covered customer management, sales and receipt handling, used item and trade-in tracking, inventory management, pricing management, repairs, payment methods, and employee access control.

It also had to support performance, security, maintainability, and future expansion. That meant we were not just building for a single transaction, but for the reality of a store that needs consistency, accountability, and room to grow.

What worked well

This project worked well because it kept pushing us back to business logic. Instead of inventing random tables, we had to ask what problem each part of the design actually solved.

The normalization process was useful because it showed how easy it is for a system to become messy if entities and relationships are not thought through properly from the start.

What I learned

This project taught me that database design is not separate from analysis. The structure of the tables depends directly on understanding the business, its workflows, and its constraints.

It also reinforced how important it is to design systems around transactional reality. In this case, the database needed to support live store operations like repairs, pricing, customer history, and inventory changes, not just report on them afterward.

Small reflection

Compared to earlier database work, this project felt more realistic because there were more moving parts and more judgment involved. There was no single obvious answer, so a lot of the value came from defending why certain relationships, boundaries, and requirements made sense.

I think this project demonstrates growth in how I approach systems. I was not just thinking about what entities belong in a diagram. I was thinking more about whether the whole thing would actually make sense if someone had to use it every day.
