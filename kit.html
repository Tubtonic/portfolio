KIT Relationship Manager

My capstone project and the best example in this portfolio of building a complete system from idea to working application.

Role
Individual project
Main stack
Flutter, Node.js, Express, Prisma, SQLite
What it taught me
How to connect a full system end to end

Overview

KIT, short for Keep In Touch, is a mobile relationship manager. The idea came from a simple real problem: people lose track of when they last reached out to someone, and relationships can slowly drift because of that. I wanted to build something that felt practical and personal instead of just another assignment app.

The project ended up becoming a Flutter mobile app connected to a Node.js and Express backend, with Prisma handling the database layer and JWT used for authentication. It gave me one place where frontend, backend, database design, authentication, and data flow all had to work together.

Why this project is here:
I chose this project because it shows an important part of my development through the CPA program, whether that was building a complete system, improving how I structure code, understanding business requirements better, or learning from earlier mistakes.

What I built

On the backend, I created a REST API running on port 4000. It handles registration, login, protected routes, user data, people records, interactions, and reminder logic.

In the database, I designed the main models around User, Person, and Interaction. Each person has a check-in interval, which feeds into reminder logic that separates people into due now and upcoming groups.

On the frontend, I built the Flutter app screens for login, dashboard, adding people, and logging interactions. I connected the app to the backend using 10.0.2.2:4000 through the Android emulator, and once a user logs in the token is stored and reused for future requests.

What worked well

One thing that worked well was building the project in stages. I got the backend stable first, then tested endpoints in Postman, then connected the mobile app after I knew the API was behaving properly.

The dashboard ended up being the strongest part of the app. Once the reminder logic was working, seeing people move into the Due Now section after an interaction was logged made the whole system finally feel real.

What was challenging

The hardest part was integration. There were times where the API worked in Postman but failed in the Flutter app because of token handling, headers, or how data was being passed around.

This project also taught me that bugs often do not live in one place. A problem could start in the frontend, but the actual cause might be in the backend route, the request format, or the data model.

What I learned

This project taught me a lot about separating responsibilities clearly. The frontend should handle display, the backend should handle logic, and the data layer should stay consistent underneath both.

I also learned that structure early on matters a lot more than it seems when you are rushing to build features. Some of the fixes I had to make later came from earlier shortcuts.

More than anything, this was the project where I felt like I was finally building a complete system instead of isolated course pieces.

How it reflects my growth

KIT shows that I can take an idea, break it into parts, build those parts, debug them, and keep working until the full flow functions properly.

It also represents a shift in how I think. Earlier in the program I was mostly focused on completing specific tasks. With this project, I had to think like someone designing and maintaining a real application.
