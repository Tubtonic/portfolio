Flag Guessing Game

An earlier C# project that still matters to me because of how much it taught me about software teamwork, game logic, and polish.

Role
Team project
Main stack
C#
What it taught me
Version control, planning, and polish matter

Overview

This was a C# flag guessing game where a flag is shown and the player has to choose the correct country from four options. It included random flag display, scoring, configurable end conditions, time limits, hints, voice recognition, result screens, graphics, audio, and documentation.

Even though it is one of my earlier projects, I still think it belongs in this portfolio because it taught me a lot and because some of the lessons from it still matter in my later work.

Why this project is here:
I chose this project because it shows an important part of my development through the CPA program, whether that was building a complete system, improving how I structure code, understanding business requirements better, or learning from earlier mistakes.

What worked well

The game itself had a lot of functionality working. It was more than just a basic multiple-choice exercise because it had scoring logic, timing, hinting, result screens, and even voice-based answer selection.

From a learning standpoint, it was also one of the first projects where I really started seeing how many little pieces have to come together before something feels complete.

What did not work well

The biggest issue was not the code itself. It was how our team handled development. I was not the owner of the repository, work was being passed around in a messy way, and one other group member handled all the commits.

Looking back, that was a bad setup and a good lesson. It showed me how important proper version control and shared workflow really are.

What I learned

I learned a lot about building game logic in C#, but also about scope and polish. Once the features were finally working, there was not enough time left to make the UI and audio feel as refined as they should have.

That taught me that getting features working is not the same thing as finishing a project properly. Time has to be set aside for refinement too.

How it reflects my growth

This project represents an earlier stage of my development, but it is still useful because it makes the later growth easier to see. It taught me teamwork lessons, version control lessons, and the importance of proper planning that I carried into later work.
