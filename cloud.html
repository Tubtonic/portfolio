Reading Nook Cloud Migration

A team cloud project where we deployed a containerized bookstore application using AWS services and an automated CI/CD pipeline.

Role
Team project
Main stack
AWS ECS, ECR, RDS, ALB, Secrets Manager
What it taught me
How containerized systems are deployed in practice

Overview

This project was based on migrating the Reading Nook inventory website into the cloud. The system included a Next.js frontend, a Flask REST API backend, and a PostgreSQL database. Our job was to design and implement the AWS side so that the system could be built, stored, and deployed through a proper cloud workflow.

What made this one valuable for me is that it moved beyond writing application code and into understanding how applications are actually hosted, connected, monitored, and deployed.

Why this project is here:
I chose this project because it shows an important part of my development through the CPA program, whether that was building a complete system, improving how I structure code, understanding business requirements better, or learning from earlier mistakes.

What the architecture included

The frontend and backend were containerized and deployed separately through Amazon ECS with Fargate, which meant we did not have to manage servers directly.

Docker images were stored in Amazon ECR, the database lived in Amazon RDS for PostgreSQL, traffic was routed through an Application Load Balancer, and credentials were stored through AWS Secrets Manager.

The project also used GitHub Actions so that new images could be built, pushed, and deployed automatically when code was committed.

What worked well

A strong part of this project was how the architecture fit together. The separation between frontend, backend, and database made the system easier to understand, and the AWS services each had a clear job.

Using managed services also made the design more realistic. RDS handled maintenance concerns better than a self-managed database, and ECS with Fargate simplified deployment compared to dealing with servers manually.

What I learned

This project helped me understand the practical side of cloud infrastructure a lot better. Before this, a lot of AWS could feel abstract. Here, it was clearer why different services exist and how they connect.

I also learned more about deployment consistency. A CI/CD pipeline sounds simple when you describe it in one sentence, but seeing it tied to containers, registries, services, and logs made it feel much more concrete.

Small reflection

This project showed me that cloud work is not just about spinning something up and saying it runs. It is about how services are separated, how secrets are handled, how traffic reaches the app, and how updates get pushed in a repeatable way.

I think it reflects my growth because I was better able to connect the application side with the infrastructure side. That matters a lot for modern development, even if my main focus is not a pure cloud role.
